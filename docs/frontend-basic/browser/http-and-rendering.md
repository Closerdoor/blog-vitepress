## 浏览器在做什么

浏览器的核心工作可以拆成两部分：

- 通过网络拿到资源。
- 解析资源并把页面渲染出来。

从职责上看，常见浏览器通常包含这些模块：

- 用户界面：地址栏、前进后退、标签页等。
- 浏览器进程：协调网络、渲染、存储等能力。
- 渲染引擎：解析 HTML 和 CSS，生成页面。
- JavaScript 引擎：执行脚本。
- 网络模块：发起 HTTP 请求、处理缓存与连接。
- 存储模块：Cookie、LocalStorage 等持久化能力。

## HTTP 是什么

HTTP 是客户端和服务端之间传输资源的应用层协议。

- 客户端发起请求。
- 服务端返回响应。
- 资源可以是 HTML、图片、JSON、文本或二进制文件。
- 协议本身是无状态的，会话通常依赖 Cookie、Session 或 Token 维持。

## URL 结构

```text
scheme://host:port/path?query#fragment
```

- `scheme`：协议，如 `http`、`https`。
- `host`：域名或 IP。
- `port`：端口，HTTP 默认 `80`，HTTPS 默认 `443`。
- `path`：资源路径。
- `query`：查询参数。
- `fragment`：锚点。

## 一次请求的大致流程

用户在地址栏输入一个 URL 后，浏览器通常会经历这些步骤：

1. 先检查本地缓存是否可直接命中。
2. 进行 DNS 解析，拿到目标服务器 IP。
3. 建立 TCP 连接；如果是 HTTPS，还要完成 TLS 握手。
4. 发送 HTTP 请求报文。
5. 服务端处理请求并返回响应。
6. 浏览器根据响应头的状态码和 `Content-Type` 决定后续动作。
7. 如果返回的是 HTML，渲染进程开始解析文档并构建页面。

常见响应处理：

- `301`、`302`：重定向，浏览器会继续请求新地址。
- `text/html`：进入页面解析和渲染流程。
- `application/octet-stream`：通常触发下载。

## 请求和响应分别包含什么

### 请求报文

- 请求方法：`GET`、`POST`、`PUT`、`DELETE` 等。
- 请求路径和查询参数。
- HTTP 版本。
- 请求头。
- 请求体。

### 响应报文

- HTTP 版本。
- 状态码和状态描述。
- 响应头。
- 响应体。

## HTTP/2 的几个变化

相比 HTTP/1.1，HTTP/2 常见改进包括：

- 二进制分帧，协议解析更稳定。
- 多路复用，多个请求可复用同一连接。
- Header 压缩，减少重复头部开销。
- 支持 Server Push，但现在实际使用已经不多。

## HTML 返回后，浏览器如何开始渲染

浏览器拿到 HTML 后，不会等整个文档下载完成才开始工作，而是边下载边解析。

核心链路一般是：

1. 解析 HTML，构建 DOM。
2. 解析 CSS，构建 CSSOM。
3. 合并 DOM 与 CSSOM，生成渲染树。
4. 计算布局，确定元素位置和尺寸。
5. 执行绘制，把像素输出到屏幕。

这也是首屏性能里最关键的一条渲染路径。

## `script` 为什么会阻塞解析

默认情况下，浏览器解析 HTML 遇到 `script` 时会暂停。

原因不是脚本“必须慢”，而是脚本可能立刻改写当前 DOM 或读取样式信息。如果继续往下解析，当前文档状态就可能不一致。

```html
<script src="/app.js"></script>
```

默认行为：

- 下载脚本。
- 执行脚本。
- 执行结束后继续解析 HTML。

## `defer` 和 `async`

```html
<script defer src="/main.js"></script>
<script async src="/analytics.js"></script>
```

### `defer`

- 下载时不阻塞 HTML 解析。
- 等文档解析完成后再执行。
- 多个 `defer` 脚本按出现顺序执行。
- 通常适合业务主脚本。

### `async`

- 下载时不阻塞 HTML 解析。
- 一旦下载完成就立即执行。
- 执行顺序不保证。
- 更适合统计、埋点、广告这类独立脚本。

## CSS 为什么也会影响渲染

CSS 本身不会修改 DOM，但它会影响脚本读取样式结果，也会决定页面能否正确显示。

常见现象：

- CSS 太晚加载，可能出现 FOUC，也就是先看到无样式内容，再闪成最终样式。
- 某些浏览器在关键样式未准备好前会延迟首屏绘制，看起来像白屏更久。

实践上通常应把关键 CSS 尽早加载。

## `DOMContentLoaded` 和 `load`

- `DOMContentLoaded`：HTML 已经被完整解析，不必等待图片、样式表等全部资源加载完。
- `load`：页面所有资源都加载完成后触发。

如果只想在 DOM 可操作后执行初始化逻辑，通常监听 `DOMContentLoaded` 就够了。

## 常见性能结论

- 减少关键资源数量。
- 缩短关键请求链路。
- 让关键资源体积更小。
- 把非关键脚本延后，避免阻塞首屏。
- 把关键样式前置，避免闪烁和白屏。

## 总结

HTTP 负责把资源送到浏览器，渲染流程负责把资源变成页面。性能优化的核心，不是记术语，而是知道哪一段在阻塞首屏、哪一种资源在拖慢用户看到内容的时间。
