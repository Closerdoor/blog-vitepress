## 总览

浏览器把页面显示出来，通常会经过这条链路：

1. 解析 HTML，生成 DOM。
2. 解析 CSS，生成 CSSOM。
3. 生成渲染树。
4. 布局。
5. 分层。
6. 绘制。
7. 分块与光栅化。
8. 合成并显示到屏幕。

## 1. 构建 DOM

HTML 解析器会边接收数据边生成 DOM，而不是等整份 HTML 下载完成后再统一处理。

- 开始标签和结束标签会被分词。
- 解析器按结构构建节点树。
- 不合法结构也会尽量被浏览器修正。

遇到同步脚本时，HTML 解析会暂停，脚本执行完成后再继续。

## 2. 计算样式

样式来源通常有三类：

- 外链样式表：`<link>`
- 页面内样式：`<style>`
- 行内样式：`style` 属性

浏览器会把这些规则整理成 CSSOM，并计算每个节点的最终样式。

这里要注意：

- 继承属性会向下传递，如字体、颜色等。
- 同一个节点最终采用的是层叠后的结果，而不是某一条单独规则。

## 3. 生成渲染树

渲染树不是 DOM 的简单拷贝。

- 只保留需要显示的节点。
- `display: none` 的节点不会进入渲染树。
- 一个 DOM 节点可能对应多个渲染对象。

渲染树描述的是“哪些内容要显示，以及它们长什么样”。

## 4. 布局

布局阶段会计算每个可见节点的几何信息：

- 位置
- 宽高
- 盒模型关系

这一步也常被叫做回流或重排。

如果页面结构复杂、节点很多，布局成本会比较高。

## 5. 分层

不是所有内容都在同一个图层里绘制。浏览器会根据元素特征决定是否分层，常见触发因素包括：

- `z-index`
- `position`
- `transform`
- `opacity`
- 某些动画效果

分层的目的，是让后续绘制和合成更高效。

## 6. 绘制

绘制阶段会把每个图层的内容转成绘制指令。

这时浏览器知道：

- 哪些区域要画背景。
- 哪些区域要画文字。
- 哪些区域要画边框、阴影、图片。

绘制不等于直接上屏，它更像是在生成“怎么画”的命令列表。

## 7. 分块与光栅化

图层通常会被切成多个图块，例如 `256 x 256` 或 `512 x 512`。

接着浏览器把图块转成位图，这一步叫光栅化。

- 光栅化通常可以并行处理。
- GPU 会参与这一步，加速位图生成和后续合成。
- 优先处理视口附近内容，有利于更快显示当前可见区域。

## 8. 合成

当图层都准备好后，浏览器会把多个图层按顺序合成为最终画面，然后提交到屏幕显示。

这也是为什么某些只发生在合成阶段的动画，性能通常更稳定。

## 重排、重绘、合成的区别

### 重排

影响布局的改动会触发重排，例如：

- 元素尺寸变化。
- 增删可见节点。
- 内容变化导致尺寸变化。
- 读取某些强制布局属性后又继续写样式。

常见代价最高。

### 重绘

只影响外观、不影响布局时会触发重绘，例如：

- `background-color`
- `color`
- `box-shadow`

重绘比重排轻，但频繁发生依然会有成本。

### 合成

如果改动只发生在合成阶段，通常不会重新布局和重绘，例如：

- `transform`
- `opacity`

这类属性更适合动画场景。

## 为什么操作 DOM 可能很慢

DOM 慢不只是因为“节点多”，更因为它会牵动整个渲染流水线。

例如下面这些操作都可能触发布局计算：

```js
const width = element.offsetWidth;
element.style.width = width + 10 + 'px';
```

如果读写交错发生，浏览器可能被迫提前刷新布局，产生布局抖动。

## 优化建议

- 批量修改 DOM，而不是频繁逐条改。
- 优先切换 class，而不是连续写多个内联样式。
- 读布局和写布局分开处理，避免强制同步布局。
- 动画优先使用 `transform` 和 `opacity`。
- 复杂节点可以先离线处理，再一次性插回文档。
- 避免在循环中反复读取 `offsetWidth`、`offsetHeight` 等属性。

## 总结

理解渲染流水线的价值，在于你能判断一次改动到底卡在布局、绘制还是合成阶段。只要知道成本落在哪一环，性能优化就会清晰很多。
