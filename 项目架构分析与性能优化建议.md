# 项目架构分析与性能优化建议

## 1. 项目架构理解

该项目是一个 Python 后端系统，核心分为两条主线。

### 1.1 第一条线：邮件聚类基线生成

- 每 5 分钟从 ES 拉取过去 5 分钟的邮件原始日志，单批约 6k~1w 条。
- 从每封邮件中提取文本特征：邮件主题、附件名、邮件中包含的域名。
- 对每封邮件做向量化，得到一个 512 维向量。
- 本地维护一个 1 小时滑动窗口，只保留最近 1 小时的邮件日志和对应向量数据。
- 每次新增最新 5 分钟数据后，删除最旧 5 分钟数据。
- 将 1 小时窗口内的数据整合后执行语义聚类，当前算法为 `DBSCAN`。
- 聚类成功的邮件经过进一步过滤后，作为风险结果传递给下游。

这一条线本质上是一个持续运行的批处理循环任务，每 5 分钟产出一批聚类结果。

### 1.2 第二条线：在线检测服务

- 每当第一条线产出一批聚类结果后，第二条线会提取其中的邮件 `md5`、域名信息以及聚类成功邮件的向量化数据。
- 第二条线启动一个 `FastAPI` 服务，对外提供邮件检测接口。
- 接口输入为邮件原始信息，例如主题、附件、附件 md5、域名、发件人等。
- 接口输出为 `result=true/false`，表示该邮件是否判定为钓鱼邮件。

检测流程分为 6 步：

1. 加载基线数据并预处理输入邮件。
2. 白名单匹配，命中则直接返回 `false`。
3. `md5` 匹配，命中基线则返回 `true`。
4. 域名匹配，命中基线则返回 `true`。
5. 对输入邮件向量化，并与基线向量一起执行语义聚类；若该邮件成功聚类，则返回 `true`。
6. 若前述流程都未命中，则返回 `false`。

### 1.3 基线更新机制补充

这里需要特别记录一个实现事实：基线更新与检测接口已经解耦。

- 基线管理使用 `COW` 模式。
- 新基线会在后台完整加载并处理完毕。
- 只有在新基线准备完成后，才会原子化替换旧指针。
- 检测接口侧只从基线管理器获取当前最新的一份只读数据。

这意味着在线检测路径本身不依赖实时构建基线，也不需要等待基线更新完成。这是一个正确且重要的设计，它已经避免了“请求路径阻塞在基线构建上”的问题。

## 2. 当前问题与瓶颈判断

当前目标是：

- 峰值每秒处理 100 封邮件。
- 单封邮件从下游发起请求到收到结果，耗时小于 2 秒。

目前观测到的现象是：

- 最慢场景通常需要 2~3 秒。
- 已尝试扩容到更大的 CPU 服务器，但收益不明显。
- 已持续调整 `torch` 线程配置。
- 已调整 `gunicorn` 的 `workers`、`threads` 等参数。
- 最慢阶段通常出现在向量化和语义聚类过程。

结合补充信息，可以得到几个比较关键的判断。

### 2.1 在线聚类规模并不大

在线请求参与聚类时，基线向量最大值设置为 50，加上输入邮件后最多只有 51 个 512 维向量一起执行 `DBSCAN`。

这说明：

- 在线 `DBSCAN` 的输入规模很小。
- 从纯算法量级上看，`DBSCAN` 本身不太像是 2~3 秒延迟的唯一核心来源。
- 需要重点怀疑的是向量化成本、数据准备成本、并发争抢、排队等待、线程过量配置等问题。

### 2.2 向量化很可能是在线路径中的核心重计算步骤

已知单条邮件向量化约为 100ms。这在低并发下可以接受，但在高并发下会迅速放大：

- 100 req/s 意味着每秒要完成大量模型推理。
- 如果每个请求内部又使用高线程数，系统会进入严重的线程争抢状态。
- 最终体现为吞吐不升反降、尾延迟上升、平均延迟也变差。

### 2.3 当前部署参数很可能存在过度并发

当前在 32 核服务器上的经验配置为：

- `workers=16`
- `threads=4`
- `torch=16`

这个组合表面上看“资源利用充分”，但对 CPU 密集型任务来说风险很高：

- Gunicorn 多 worker 已经提供了进程级并发。
- 每个 worker 再开多个线程，会进一步叠加调度竞争。
- 每个请求内 `torch` 再启用较高线程数，会形成第三层并发。

这类配置常见结果是：

- 单请求并没有稳定变快。
- 系统总线程数远超物理核数。
- 上下文切换增多。
- CPU cache 命中率下降。
- `p95/p99` 延迟显著恶化。

因此，问题很可能不是“CPU 不够多”，而是“并发模型不适合 CPU 推理场景”。

## 3. 对当前架构的分析结论

### 3.1 当前系统本质上由两种负载构成

第一条线是准实时批处理负载：

- 拉取日志
- 维护滑动窗口
- 语义聚类
- 产出风险基线

第二条线是在线低延迟检测负载：

- 请求驱动
- 需要稳定响应时间
- 对高并发敏感

这两种负载的最优调度方式通常不同，因此你已经做的“基线更新与接口解耦”是合理的。

### 3.2 在线检测链路中，真正值得重点关注的是 CPU 推理并发模型

当前在线路径中，前几步都是轻量匹配：

- 白名单匹配
- `md5` 匹配
- 域名匹配

真正重的步骤主要是：

- 输入邮件向量化
- 小规模 `DBSCAN`

其中，按现有数据判断，向量化更像主要开销来源，而 `DBSCAN` 更像在高并发下被整体系统抖动放大的次要放大项。

### 3.3 高并发场景下，尾延迟比单次计算耗时更值得关注

在接口性能测试中，如果单次向量化约 100ms、在线聚类规模仅 51 条，但总延迟能达到 2~3 秒，则通常意味着：

- 请求存在排队等待
- 推理线程之间互相抢占 CPU
- 进程和线程层面有过量并发
- 部分请求被调度延后，形成长尾

因此，优化方向应优先落在“降低争抢、控制并发、拆分耗时埋点、找到真正长尾来源”，而不是继续单纯增加 CPU 数量。

## 4. 在当前约束下的优化建议

当前约束是：

- 现阶段不适合做大规模架构改造。
- 在线相似性判断仍需继续采用 `DBSCAN`。

在这个前提下，更可行的优化策略如下。

### 4.1 优先做精细化耗时埋点

建议先将单次请求拆解为以下阶段并独立记录耗时：

- 请求进入服务到进入业务逻辑
- 白名单匹配
- `md5` 匹配
- 域名匹配
- 向量化
- 向量预处理
- `DBSCAN`
- 最终结果封装与返回

至少要观测：

- 平均耗时
- `p95`
- `p99`
- 高并发压测下各阶段耗时变化

只有拆开之后，才能区分以下几种情况：

- 真正慢的是向量化
- 真正慢的是聚类前的数据处理
- 真正慢的是锁等待或队列等待
- 真正慢的是服务级线程争抢

### 4.2 收敛 Gunicorn 与 torch 的线程策略

对 CPU 密集型场景，更推荐：

- 降低 `threads`
- 让 `workers` 承担主要并发
- 控制 `torch` 的线程数

建议优先测试以下思路：

- `threads=1`
- `workers` 在 `8`、`12`、`16` 三档测试
- `torch` 线程在 `1`、`2`、`4` 三档测试

同时建议统一约束以下环境变量，使它们与 `torch` 配置保持一致：

- `OMP_NUM_THREADS`
- `MKL_NUM_THREADS`
- `OPENBLAS_NUM_THREADS`
- `NUMEXPR_NUM_THREADS`

这一方向的目标不是让单请求吃满更多核，而是避免所有请求都在争抢过多核。

### 4.3 为向量化阶段增加受控并发

即使接口可以接收很多请求，进入模型推理的并发数也不应无限制增长。

建议：

- 给向量化步骤增加一个并发上限
- 让过量请求在服务内部短暂排队，而不是同时冲进模型推理

这听起来像“降低并发”，但在 CPU 推理场景下往往能显著改善：

- 平均延迟
- 长尾延迟
- 实际可持续吞吐

### 4.4 优化 DBSCAN 的输入准备，而不是纠结算法本身

由于在线聚类样本最多只有 51 条，因此真正需要优化的重点更可能是“聚类前后的数据组织方式”。

建议：

- 基线向量常驻为连续的 `float32 numpy` 数组
- 若需要归一化，则基线向量在基线构建阶段预处理完成
- 请求到来后只拼接 1 条输入向量，避免重复的 `list -> numpy -> list` 转换
- 尽量减少 Python 层循环和对象复制

如果当前 `DBSCAN` 使用的是余弦距离，也可以评估是否在不改变算法结论的前提下，减少重复距离计算和预处理动作。

### 4.5 尽可能提高前置短路命中率

当前的前置判断顺序是合理的：

1. 白名单
2. `md5` 匹配
3. 域名匹配
4. 向量化 + 聚类

建议重点关注：

- 白名单是否足够完备
- 域名规则是否还能前移或增强
- 是否能在进入向量化前增加一些廉价过滤条件

目标是尽量减少进入“向量化 + 聚类”重计算路径的请求比例。

### 4.6 评估向量化结果缓存

如果邮件场景中存在明显重复，例如：

- 相同主题
- 相同附件名
- 相同附件 md5
- 相同域名组合
- 模板化邮件内容

则可考虑：

- 对规范化后的邮件特征做 embedding 缓存
- 甚至对最终判定结果做短 TTL 缓存

这类优化不一定改变整体架构，但在真实业务流量中常常非常有效。

## 5. 对现象的核心解释

结合现有信息，我对当前问题的核心解释如下：

### 5.1 问题未必是算力绝对不足

在 32 核乃至更大 CPU 机器上，单条 100ms 左右的 embedding 推理如果没有获得预期收益，通常不意味着机器不够强，而是：

- 并发粒度不合理
- 线程数过多
- 每层都在并发
- 系统进入过度调度状态

### 5.2 在线 DBSCAN 更像被放大的次要成本

由于在线聚类样本只有 51 条左右，`DBSCAN` 自身的计算量不应成为主要的秒级瓶颈。更大的可能是：

- 向量化阶段已经产生显著等待
- 聚类前数据准备有重复转换
- 总耗时里包含排队与争抢时间

### 5.3 你真正需要优化的是“稳定性”

当前目标是峰值 100 req/s 且单封邮件 < 2s，这本质上要求系统在高并发下保持稳定的服务曲线。对于这类目标，通常比“把单次推理跑到最短”更重要的是：

- 控制推理阶段并发
- 降低线程过量竞争
- 保证请求路径只读、轻量、可预测
- 把长尾压下来

## 6. 后续可执行的压测与调优方向

如果后续进入实操调优阶段，建议采用矩阵式压测，而不是继续凭经验单独调整某一个参数。

### 6.1 优先测试的参数组合方向

可从以下组合开始测试：

- `workers=8, threads=1, torch=4`
- `workers=12, threads=1, torch=2`
- `workers=16, threads=1, torch=1`
- `workers=8, threads=1, torch=2`

压测时关注：

- 成功请求数
- 平均延迟
- `p95`
- `p99`
- CPU 利用率
- 上下文切换情况
- 请求在各阶段的耗时占比

### 6.2 重点识别三类问题

通过压测结果，需要优先判断以下三类问题中的哪一类最突出：

1. `torch` 内部线程数过高导致的推理争抢
2. Gunicorn worker/thread 组合不合理导致的服务级排队
3. 请求路径中实际存在锁等待、对象复制或数据准备开销

## 7. 若未来允许做更大改动，可考虑的演进方向

虽然当前阶段不适合做大规模改造，但从长期看，仍可以记录几个可演进方向：

- 将在线相似性判断从“重跑聚类”演进为“相似度检索 + 阈值判断”
- 将 embedding 推理独立成受控的本地进程池或独立服务
- 将风险簇的代表向量、中心向量或近邻索引提前离线构建

这些方向不属于当前必须执行的内容，但从系统演进角度看，未来会更适合高并发低延迟场景。

## 8. 最终结论

当前项目架构本身并没有明显的方向性错误，尤其是基线更新采用 `COW` 并与在线接口解耦，这一点是合理且成熟的。

现阶段性能问题更可能来自以下几点的组合：

- CPU 推理阶段的线程配置过于激进
- Gunicorn 与 torch 的并发层次叠加过多
- 高并发下出现排队、争抢和长尾放大
- 在线 `DBSCAN` 的前后数据准备存在可优化空间

因此，在“不改大架构、继续使用 DBSCAN”的约束下，最务实的优化方向应当是：

1. 做精细化埋点，找出真实耗时分布。
2. 收敛 worker/thread/torch 的并发模型。
3. 对向量化阶段增加受控并发。
4. 优化 DBSCAN 前后的数据准备与内存组织。
5. 尽量提升前置短路规则的命中率。

如果这几项做完，通常比继续单纯增加 CPU 资源更可能带来稳定收益。

## 9. 关于 detect 方法是否需要异步化

在后续讨论中，针对在线检测主流程是否要改成异步实现，也形成了比较明确的判断。

### 9.1 核心结论

当前 `detect` 主流程不建议为了“提速”而改造成 `async`，也不建议单纯使用 `asyncio.to_thread()` 给整个检测逻辑再包一层。

原因是当前检测主路径的主要工作属于 CPU 密集型计算，而不是 I/O 密集型等待：

- 白名单匹配
- `md5` 匹配
- 域名匹配
- 向量化
- `DBSCAN`

对于这种类型的逻辑：

- `async` 本身不会让 CPU 计算变快
- `asyncio.to_thread()` 的主要作用是避免阻塞事件循环，而不是提升吞吐
- 如果当前系统已经存在 worker、thread、torch 多层并发，再额外引入线程包装，可能只会进一步增加调度成本

### 9.2 更适合当前场景的原则

更适合当前项目的原则是：

- 检测主流程保持同步思维
- 不为了形式上的“异步化”而修改核心检测链路
- 优先优化 CPU 推理并发模型，而不是协程结构

如果接口层使用 `FastAPI`，那么真正要避免的是：

- 在 `async def` 路由中直接执行长时间 CPU 计算，导致事件循环被阻塞

但这个问题的解决重点仍然不是“给 detect 套一层 async”，而是：

- 明确在线检测属于 CPU 密集型任务
- 让它在合适的执行模型中运行
- 控制并发，降低争抢

### 9.3 方案级建议

更推荐的结构是：

- `detect` 作为同步方法，负责完整检测流程
- 接口层尽快拿到 `result=true/false`
- 与返回结果无关的后处理动作从主路径分离

也就是说，真正值得异步化和解耦的，往往不是检测本身，而是检测完成后的附属处理，例如日志记录、埋点落盘、审计信息写入等。

## 10. 关于 resultSession 日志的处理方式

后续讨论里补充了一个非常关键的细节：

- 每次检测都会有一个 `session`
- 该 `session` 会统计检测过程中的一些信息
- 检测结束后，会拿到一个 `resultSession` 字典
- 接口只需要从中提取最终结果 `result=true/false` 作为响应
- 其余日志信息，例如命中的检测场景、各步骤耗时等，需要写入日志

这个设计方向本身是合理的，但处理方式需要和检测主链路解耦。

### 10.1 建议的职责拆分

建议把一个请求的处理拆成两个部分：

1. 检测主流程
2. 日志/审计后处理流程

其中：

- 检测主流程负责尽快完成判定
- 日志后处理负责记录细节信息

前者应优先保证响应时间，后者应尽量不要阻塞接口返回。

### 10.2 resultSession 的使用建议

`resultSession` 更适合作为“单次请求上下文”的容器，用来汇总本次请求的诊断信息，例如：

- 请求标识
- 命中的检测阶段
- 每一步的耗时
- 最终结果
- 命中的关键特征
- 发生的异常或降级信息

这个对象建议保持为“每请求独立”的局部数据结构，而不是共享状态。这样可以避免并发下的数据污染，也便于在请求结束后统一交给日志链路处理。

### 10.3 日志写入不要阻塞接口主路径

对于 `resultSession` 的日志信息，不建议采用“先同步写日志，再返回接口结果”的方式。更合理的做法是：

- 检测主流程完成后，先返回 `result`
- 将 `resultSession` 中需要落盘或上报的部分，交给异步日志链路处理

这里的“异步”重点是“从请求主路径中解耦”，而不是要求整条检测链路都改成 `async`。

### 10.4 更推荐的异步日志方式

在这个场景下，比“每次请求结束后额外起一个 `to_thread` 写日志”更稳的方式通常是：

- 主路径只负责生成日志事件
- 将日志事件快速投递到一个内存队列
- 由后台专门的日志消费者统一写文件、落库或上报

这种方式的优势在于：

- 不会把日志 I/O 混入检测主路径
- 不会为每个请求再额外制造一层线程调度
- 更容易控制日志写入速率
- 更适合高并发场景下的稳定性要求

### 10.5 伪结构建议

从结构上看，可以抽象为以下三段：

1. 检测函数：生成 `result` 和 `resultSession`
2. 接口层：立刻返回 `result`
3. 日志链路：异步消费 `resultSession` 中需要记录的字段

也就是说，接口层只做两件事：

- 从 `detect` 获取判定结果
- 将需要记录的日志信息交给后台处理

而不是让接口层自己承担完整日志写入过程。

### 10.6 日志内容应尽量收敛

在高并发场景下，不建议把整个 `resultSession` 原样、全量、无筛选地写入日志。更推荐只保留真正有诊断价值的字段，例如：

- `request_id`
- 最终判定结果
- 命中阶段
- 各阶段耗时
- 关键命中特征
- 错误信息
- 时间戳

这样可以减少：

- 序列化开销
- 日志体积
- I/O 压力

### 10.7 这一部分与主流程异步化的关系

因此，需要区分两类“异步”：

- 检测主流程异步化
- 日志后处理异步化

对当前项目，更值得做的是第二种，而不是第一种。

换句话说：

- `detect` 本身不应为了 async 而 async
- `resultSession` 的落日志过程应尽可能从主路径剥离

## 11. 现阶段的补充结论

结合新增讨论，当前阶段可以把结论进一步收敛为：

1. 基线更新已经通过 `COW` 与检测接口解耦，这部分设计是合理的。
2. 在线检测链路中的主要风险仍然是 CPU 推理并发模型，而不是基线更新方式。
3. `detect` 主流程不应为了“看起来异步”而改造成 `asyncio.to_thread()` 风格。
4. 真正应该异步化的是与响应结果无关的后处理部分，尤其是 `resultSession` 的日志记录。
5. 更理想的接口路径应当是：同步检测、快速返回、日志异步落地。

在当前约束下，这种方式通常比把整个检测流程协程化更稳，也更符合 CPU 密集型检测服务的特点。
